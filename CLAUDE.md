## Conventions & invariants
- **QueryStatus:** always use the exact `QueryStatus` enum strings (e.g. `"Partial Requested"`, `"Revise & Resubmit"`). Never camelCase or ad-hoc variants.
- **Undo:** undo must delete the original activity records this action created/modified ‚Äî never append compensating entries.
- **Response counting:** each query counts as at most one response, regardless of pipeline stage.
- **Package slots:** the three `SubmissionPackage` slot fields (`queryLetterVersionId`/`synopsisVersionId`/`samplePagesVersionId`) use `""` (empty string) to mean an **unfilled** slot. `isValidPackage` (firestore.rules) requires all three keys **present**, so a package write MUST send `""` for an empty slot ‚Äî never omit the key. The sentinel + `isSlotFilled` live in `src/lib/packageMetrics.ts` (`UNFILLED_SLOT`); the Package Builder composer and the future attach-to-query flow both go through it.
- **Legacy top-bar conventions (now MOBILE SLIM BAR only ‚Äî the desktop bar is retired, see Navigation & shell architecture):** the "canonical nav active treatment" (black `#241c15` links, pink pills `#f3e3dc` hover / `#f5e2da` active, `NAV_INK`/`NAV_PILL_HOVER`/`NAV_PILL_ACTIVE` in `Nav.tsx`), the `--content-max` (1440px) inner-content cap, and the locked 77px bar height / type sizes all describe `Nav.tsx`, which now renders below `md` only. The desktop rail has its own treatment (`--navpill` pill, burgundy active text ‚Äî per the appshell mockup). The 45px mini-bar branch was deleted in the rollout. `NavSearch` placeholder styling (`.nav-search-field::placeholder`, Playfair 500 16px) is shared by the rail search ‚Äî still canonical. Page content stays full width (Nick's call); reference `--content-max`, never hardcode 1440.

## Dashboard ‚Äî "Over to you" & the nudge flow
- **"Over to you" container** (`src/components/dashboard/OverToYou.tsx`): a height-capped, internally-scrolling MountCard with two zones ‚Äî a pinned **"To-do list"** header (serif title + pulsing dot + burgundy urgent-count pill) above the urgent/overdue rows, then a **"When you have a moment"** recommended section whose buckets expand inline, and a footer hint shown (via IntersectionObserver on the Zone-B header) only while that section is off-screen. - **Nudge flow:** the `nudge_overdue` row's button opens `NudgeModal` (`src/components/NudgeModal.tsx`, inside the Form-11 `FormShell`) with `CheckBackSlider` (`src/components/forms/CheckBackSlider.tsx`, a day-then-week sibling to `WeekSlider`). Logging a nudge runs `logNudge` ‚Äî an **isolated path** (`src/lib/db.tsx` + the pure `src/lib/logNudge.ts` builder); do NOT fold it back into `dismissTask`.
- **logNudge field mechanics (smallest blast radius):** writes a **non-status** `NUDGE_SENT` activity (no `resultingStatus`, so `recomputeQuery` ignores it) to the top-level `activities` feed ‚Äî description must start `"Nudge sent to {agent} at {agency}"` (drives the timeline clock glyph + pill), with the check-back + optional note in `details` (new activity fields are silently denied by the Firestore allowlist). Sets `nudgeDate` + `lastNudgeSentDate`, and hides-and-resurfaces the task via a **custom-date** `DismissedTask` (`resurfaceDate` = check-back). It **never** touches `status` / `responseDeadline` and never counts as a response.
- **Derived "Follow-up reminder" timeline node** (`QuerySlideInPanel`): a non-stored projection dated from `nudgeDate`, mirroring the future-only "Response Expected" node (both render only while their date is in the future). Never enters the log, `recomputeQuery`, the response count, or the deletable list.

## Agent location / territory
- **Model:** the ONLY stored location data is `agent.country` (ISO 3166-1 alpha-2, e.g. `"GB"`) + optional `agent.city`, and `user.homeCountry` (ISO alpha-2). Everything else ‚Äî the flag, the display name, the home-vs-foreign distinction, any market count ‚Äî is **derived at read time** from those codes. Nothing territory-derived is stored on the agent, and **nothing is denormalised onto queries** (`recomputeQuery` is inert to location ‚Äî recon-confirmed).
- **ISO alpha-2 is canonical.** Legacy full-name values (the name-based `COUNTRIES` in `agentOptions.ts` + the undeployed v12 Edit Agent drawer) are **tolerated at read time only** ‚Äî `territory.ts` reverse-maps them to codes; consumers migrate to writing codes in Prompt 2.
- **Never `null`/`""`.** "Not set" always means the **key is omitted**. The `isValidAgent`/`isValidUser` rules reject a literal null (`data.x is string`). The signup seed (`homeCountrySeed()`) and quick-add both omit rather than write empty.
- **Single source of truth = `src/lib/territory.ts`** (Prompt 1): `COUNTRIES_ISO`, `QUICK_PICKS` (`GB US IE CA AU NZ`), `countryName`, `flagFor` (‚Üí `flag-icons` class `"fi fi-gb"`), `isHomeMarket`, `detectHomeRegion` (browser locale region + tz fallback, **never IP**), `getHomeCountry` (stored ?? detected ?? `GB`), `homeCountrySeed`. Flags = the whole `flag-icons` SVG set, served as static assets (CSS-referenced, out of the main JS chunk), addressable by ISO code ‚Äî **not** emoji (Windows renders flag emoji as letter-pairs).
- **`homeCountry`** seeded silently from the browser locale at signup, no onboarding step, no backfill (`getHomeCountry` covers absent users at read time), editable later in settings (Prompt 2/3).
- **Rules edited, NOT deployed.** The `homeCountry` clauses (`isValidUser` + the user-update allowlist) ride the **same `firestore:rules` deploy** as the already-in-file-but-undeployed `country`/`city` agent allowlist ‚Äî **nothing location-related works in prod until that deploy lands.** Claude Code must not deploy.
